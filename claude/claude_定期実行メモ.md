## 2026/05/07 Claude Code の3種類の定期実行方法があるっぽい

- 3種類のスケジューラー
  - /loop: 
    - 実行環境 = ローカル(セッション内)
    - 実行条件 = セッションを開いてる必要あり。セッション終了で消失。繰り返しタスクは3日で自動失効。
    - 適した用途 = セッション内でのbackgroundのpooling的なタスク(ex. デプロイ監視, ビルド確認, 長時間タスクの完了待機など)
  - Claude Desktop appのスケジューラー:
    - 実行環境 = ローカル
    - 実行条件 = セッション不要(各実行で新しいセッションが起動)。Desktopアプリが起動しており、PCがスリープしてない必要あり。
    - 適した用途 = 微調整が発生してるタスク、PDCAサイクル中の作業
    - その他備考:
      - Claude Desctop Appが起動し、かつPCがスリープしてない必要があるため、完全な放置はできない。**しかし、この「手が届く距離」が利点でもある**。
        - スキルの出力内容を確認し、プロンプトの調整やワークフローの改善をその場で行える。
        - **自動化を構築する初期段階では、この「手が届く距離」でのFBループが不可欠!!**
  - Anthoropicクラウド環境でのCloudスケジューラー
    - 実行環境 = Anthropicクラウド環境
    - 実行条件 = PCの起動状態に依存しない。最小実行間隔は1時間。
    - 適した用途 = 安定稼働してるタスクの完全な自動実行。
    - その他備考:
      - 定時になったら自動で実行され、結果が通知される。
      - 以下の条件を満たすタスクがCloudスケジューラーに適してると思われる:
        - スキルのワークフローが安定しており、**手動介入がほぼ不要**。
        - 出力フォーマットが固定されている。
        - **実行結果の確認が事後で問題ない。**
        



- なんでClaudeのスケジューラーが強力なのか?? 
  - 他のcronやタスクスケジューラーと異なるのは、**MCPコネクタやスキルを理解した状態で定期実行できる点**!
    - 普通のcronジョブ (=サーバやPCで決まった時刻に処理を自動実行する仕組み)の場合は、Slack・Notion・Google CalendarそれぞれのAPIに対して、エンジニアが個別に接続設定をコーディングする必要がある（通常は数時間〜数日の作業になる）。
    - 一方でClaude Codeのスケジューラーの場合はMCPとして接続済みのサービスをそのまま利用できる!!
      - -> よって、例えば以下のような複数サービスを横断するワークフローを、自然言語のプロンプト1つで定期実行できる!!
        - 1. Google Calendarから予定を取得
        - 2. Slackの特定チャンネルからメッセージを検索
        - 3. NotionのDBを更新
        - 4. GithubのPR情報を取得する
        - 5. それらの情報を組み合わせてレポートを生成し、Slackに投稿する
    - 結果として、**個別のAPIクライアントをコーディングする必要がなく、認証の管理も不要。この手軽さが自動化のハードルを大幅に下げている**!!

- (ちなみにブログ執筆されてたクラシルの方は、DesktopとCloudの2つを使うことが多いとのこと)
  - どんな提携業務を実際に自動化したのかの例達:
    - 1. 1on1事前準備の自動化 -> 1on1直前に慌ててメンバーの情報を収集する作業がなくなった!!
      - 平日毎朝9時にDesktop スケジューラーが起動し、Google Calendarからその日の1on1予定を検索。
      - 対象メンバーが見つかると、Slack・Notion・GitHubから過去2週間の活動を収集し、活動のサマリーを整理したドキュメントを自動生成。
      - 使用MCP: Google Calendar, Slack, Notion, GitHub
    - 2. MTG後の議事録自動生成
      - 定時MTGの話。
      - NotionAIで録音した議事録を取得し、議事録をMarkdown形式に整形してslack上に投稿。
      - 使用MCP: Slack, Notion
