## 導入: 本論文のモチベーション

- コンテンツのRecSysは、現代のオンラインサービスにおいてめちゃ重要!
  - 根底にある考え方 = パーソナライズされたユーザ体験 → サービスに対する長期的なユーザエンゲージメントを高める。また、付随的な利益としてサービスの他の側面(ex. 収益)も向上させる。
- 従来のRecSysの問題点: CTRだけではダメだよね!! 
  - 従来のRecSysでは、CTRや明示的なユーザ評価値などの単純なユーザFBシグナルが、ユーザ興味や満足度を定量化するために使われてきた。
  - ただ明示的なユーザ評価値は、基本的に集まりづらい。
  - よってCTRが広く使われる!
  - でも、暗黙的なユーザ興味シグナルとしてのCTRは、クリック後のユーザ満足度を捉えられない。(ex. 誤クリックや、クリックベイトなども含まれる!)
- ユーザ満足度を表すシグナルとして、アイテム単位の滞在時間(item-level dwell time)が有望!!
  - **これはクリックベースのシグナルを補完or置き換えるもの。** 
    - (実運用では補完してるケースが多い気がする...!!:thinking:)
- とはいえ、プロダクトで滞在時間を利用することは多くの課題がある!!
  - まずいい感じに測定するのがむずい。
  - さらに、滞在時間の性質を理解して有効活用する必要がある!!
- 本論文の目的: **滞在時間をRecSysで有効活用するための方法論を提案 & 実例を共有すること!!**

## 滞在時間のちゃんとした測定方法!

- Webブラウザのユーザ活動データから滞在時間を正確に計算するのはかなり難しい問題。
  - ex. **現代のブラウザはマルチタブインターフェース**を持つため、ユーザは複数のタブを同時に開いて切り替えることができる。
    - この設定では、ユーザがどのタブに注意を向けているかを正確に把握するのは難しい。
  - (モバイルアプリの場合は、まあそこまで難しくはないのかな...??:thinking:)
- 本論文では、滞在時間を導出するための**2つの補完的な方法**を提案:
  - 1つ目: クライアントサイドのログを使う方法
  - 2つ目: サーバーサイドのログを使う方法
  - 両者を補完的に使うことを提案してる。
- 1つ目: クライアントサイドのログを使う方法
  - 4種のイベントがある:
    - DOM-ready(ページのDOMが完全に読み込まれたとき)
    - Focus(ユーザが読み始めたとき)
    - Blur(ユーザが別タブに移動したとき)
    - Beforeunload(ユーザがページを離れる直前)
  - 主にFocusしてる区間だけを足し合わせて滞在時間を計算するというシンプルな方法。
    - t1 = DOM-readyページ読み込み
    - t2 = Focus(ユーザがタブにフォーカス)
    - t3 = Blur(ユーザが別タブに移動)
    - t4 = Focus(記事に戻る)
    - t5 = Beforeunload(ページを閉じる)
    - この場合の滞在時間 = (t3 - t2) + (t5 - t4)
