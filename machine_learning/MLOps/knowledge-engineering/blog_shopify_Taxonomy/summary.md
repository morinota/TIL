## 文書 1: blog_shopify_taxonomy.md (ICLR 2025 Expo 要約) — **必読**

ICLR 2025 で発表された **Global Catalogue** の技術概要。Shopify の中核戦略を 8 章で解説。

### I. Introduction (L9–37)

「**150 ドル以下の持続可能なランニングシューズを見せて**」という AI 時代の問いに答えるには、構造化・標準化された商品データが必須。Shopify は数百万のマーチャントが数十億の商品を独自形式で記述する → 機械理解には致命的な多様性。

### II. 問題: 断片化したデータと発見の課題 (L39–86) — 必読

5 つの課題:
- **非構造化データ**: 説明文の自由記述、長さも不揃い
- **スキーマ異質性**: 各マーチャントが独自キーを定義
- **データ品質と希薄性**: 誤字・欠損値・誤分類・無関係なコンテンツ
- **マルチモーダリティ**: 商品情報がテキスト/画像/動画に分散
- **多言語**: 大陸ごとに違う言語と用語

この断片化により、セマンティック検索が困難・ファセットが不十分・重複リスティング・関連性の低い結果。AI 時代にはさらに深刻化。

### III. 解決策: 4 層の Global Catalogue (L88–228) — **必読**

| 層 | 役割 |
|---|---|
| **1. Product Data Foundation** | 1日1000万件以上の商品更新をストリーミング処理。スキーマ進化システム + 変更データキャプチャ |
| **2. Product Understanding** | 非構造化データを標準メタデータに変換 (分類・属性抽出・画像理解・タイトル標準化・説明分析・レビュー要約) |
| **3. Product Matching** | 異なるマーチャントが売る同一商品を識別 (LSH + embedding clustering + 決定的マッチ + edge pruning + 二部グラフ) |
| **4. Reconciliation** | 商品クラスタの正規化レコード生成 (属性マージ・バリアント標準化・コンテンツ集約) |

**重要なアーキテクチャ判断**: 各エンティティ (媒体/商品/バリアント) ごとに **マルチタスクのファインチューニング VLM** を使う (タスクごとに別モデルではない)。タスク間の相互依存性 (カテゴリ推論がテキスト要約の文脈になる、要約が分類を洗練する) を活かす。

**Shopify Product Taxonomy** はオープンソース。LLM が新しい属性/カテゴリ要件を継続的に特定し、自動 + ヒューマンレビューで検証。属性は taxonomy ノードにリンクされ、階層的に伝播。

### IV. ファインチューニングと選択的フィールド抽出 (L230–267) — 必読

OpenAI/Gemini の API は規模に対して高すぎる → **オープンソース VLM をファインチューニング** (LlaVA 1.5 7B → LLaMA 3.2 11B → 現行の **Qwen2VL 7B**)。各世代で精度向上 + GPU 削減。

**選択的フィールド抽出 (Selective Field Extraction)** が肝:
- 全フィールド予測の訓練だと推論時の汎化能力が落ちる
- **訓練時にランダムにフィールドサブセットを選択** して予測させる
- 推論時の柔軟性が増し、再訓練不要
- **中央値レイテンシが 2 秒 → 500 ms**、GPU 使用量 **-40%**

### V. データ生成と継続的改善 (L269–360) — 必読

**自動アノテーションパイプライン**:
- 各タスク (例: カテゴリ分類) で **複数の LLM エージェントが独立してアノテート**
- **テストサンプル**: 人間がラベルを確定 (ゴールドラベル)
- **訓練サンプル**: **LLM 仲裁者 (arbitrator)** が最良の提案を選択、必要時に人間にフォールバック

**3 カテゴリの評価メトリクス**:
| カテゴリ | 内容 |
|---|---|
| **Task-specific** | 階層各レベルでの P/R (分類)、accuracy (属性) |
| **LLM judge** | 生成フィールド (タイトル/説明) を合成判定者がガイドラインに従って評価 |
| **Instruction** | **Field compliance rate** (要求フィールドのみ出力する頻度) + **Field invariance rate** (要求スキーマ変更でも回答が一貫する率) |

**アクティブラーニングループ**: production の低品質推論を LLM judge がフラグ → 人間レビュー → 再訓練。トークン確率の低い (= モデルが不確実な) サンプルを再訓練に投入。

### VI. スケーリングとデプロイ (L362–382) — 推奨

**1 日 4000 万 LLM コール / 160 億トークン**。最適化:
- **Triton inference server** + GPU フリート
- **Kafka ベースの Dataflow streaming**
- **FP8 量子化**
- **KV cache**
- **選択的フィールドプロンプティング** (各サーフェスが必要フィールドのみ要求)

### VII. 影響と将来 (L384–436) — 推奨

統合先: マーチャント管理画面 (リアルタイム提案) / 検索・推薦 / 埋め込み生成 / 会話型コマース / マルチチャネル統合。

下流の利点:
- 検索: 標準化メタデータで多様で関連性の高い結果 (long-tail も拾う)
- パーソナライズ: ユニファイドカタログ + 標準属性
- 会話型コマース: 構造化コンテキストとして AI アシスタントを支える
- マルチチャネル統合: オープンソース taxonomy で他社/他チャネルとも相互運用可能

**将来課題**:
- スケーラビリティ × 精度 × レイテンシの均衡
- **ユニファイド vs マルチモデル** の検討 (現在はエンティティ別、統合への移行を探索中)
- グラフベース推論 (LLM × グラフ)
- アクティブラーニングと動的再訓練の継続

### VIII. Conclusion (L438–457) — 推奨

スケールでの製品理解は、マルチモーダル LLM + 慎重なシステム設計の組合せで可能。AI 時代には構造化された信頼できる製品データがますます重要に。

## 文書 2: evolution_product_classification.md (Shopify Engineering Blog) — **必読**

製品分類システムの進化を歴史的にたどる engineering blog。文書 1 と相互補完的で、こちらは推論最適化や訓練データ作成の詳細に踏み込む。

### 製品理解への旅 (L22–121)

進化の段階:
- **2018: 基本分類** — TF-IDF + ロジスティック回帰。シンプルな例には効くが、規模/多様性で破綻
- **2020: マルチモーダル** — 画像+テキスト統合。テキストか画像のいずれかが曖昧でも理解可能に
- **2023〜: 包括的理解の必要性** — カテゴリだけでは不足。5 つの要件が浮上
  1. カテゴリより細かい製品理解
  2. プラットフォーム全体での一貫した taxonomy
  3. カテゴリに応じた意味のある属性抽出
  4. 簡易商品説明・タグ・特性などのメタデータ
  5. コンテンツ安全性と信頼性
- **現在: VLM × 構造化 taxonomy**

### Shopify Product Taxonomy (L82–104) — 必読

- **26 のビジネス分野**にまたがる
- **10,000 以上のカテゴリ**と **1,000 以上の属性**をマッピング
- 4 つの特徴:
  - **階層的分類**: 例 `Furniture > Chairs > Kitchen & Dining Room Chairs`
  - **カテゴリ固有の属性**: カテゴリごとに関連する属性セット
  - **標準化された値**: 表記ゆれを防ぐ事前定義 (一貫性) + カスタム属性 (柔軟性)
  - **クロスチャネル互換性**: 他プラットフォームとの cross walks

### Vision Language Models の役割 (L106–121)

- **マルチモーダル理解**: 画像とテキストを別々ではなく統合的に
- **ゼロショット**: 未見の商品も広範な知識で分類可能
- **自然言語推論**: 複雑なリストから豊富なメタデータを抽出
- **文脈理解**: 商品が何かだけでなく、用途/スタイル/特性まで

### 推論最適化 (L136–225) — 必読

- **FP8 量子化**: GPU メモリ削減 + 精度維持 + バッチ効率
- **In-Flight Batching (Nvidia Dynamo)**: 固定バッチではなく到着パターンに応じて動的にグルーピング → GPU アイドル時間最小化
- **KV Cache**: 注意パターンの再利用
- **2 段階予測パイプライン**:
  1. カテゴリ予測 + 簡易説明生成
  2. **カテゴリ文脈を使った属性予測** (1 の出力に依存)
- カテゴリと属性の予測は **両方成功必須**。部分失敗で自動再試行

### 訓練データ構築 (L227–252)

**マルチ LLM アノテーションシステム**:
- 複数の LLM が独立してアノテート → カバレッジとロバスト性を最大化
- 不一致時は **専用の仲裁システム**が中立判断
- **人間の検証層**: 複雑なエッジケース・新製品タイプの戦略的レビュー
- **品質監査**を定期的に実施

### 実績と将来 (L255–298)

- **マーチャントの予測カテゴリ受け入れ率 85%**
- **1 日 3000 万予測**、階層的 P/R は旧 NN アプローチの 2 倍
- 今後: **木構造 → DAG 構造**への移行 (商品ごとに複数の分類パス可能)、より細かいメタデータ抽出
