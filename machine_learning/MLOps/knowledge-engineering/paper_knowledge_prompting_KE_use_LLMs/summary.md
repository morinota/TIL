## Abstract / 1. Introduction (L1–127) — 推奨

KE (Knowledge Engineering) は KG を構築・維持するプロセス。LLM の登場で実践が変わるが、効果的な使い方はまだ初期段階。著者らはハッカソンで「研究者×LLM」の相互作用を調べた。

**5 つのリサーチクエスチョン**:
- RQ1: KE タスクで LLM を使うときの主な課題は?
- RQ2: 知識エンジニアは LLM 出力をどう評価する?
- RQ3: LLM を実践に組み込むためにどんなスキルが必要か?
- RQ4: 責任ある AI 利用への意識はどの程度か?
- RQ5: LLM への信頼と採用に影響する要因は?

**3 つの貢献**:
1. 知識エンジニア×LLM の相互作用に関する初期のユーザースタディの一つ
2. LLM 利用時の強みと課題を特定
3. **KG cards (KGカード)** を提案 — Data Cards / Model Cards の発想を KG に移植

**読みどころ**: 5 つの RQ が論文全体の地図。ここを押さえれば 4・5 章を読みやすい。

## 2. Background and Related Work (L129–411)

### 2.1 Knowledge Engineering (L134–184) — 任意

KG 構築のアクター (ドメイン専門家・知識エンジニア・オントロジーエンジニア)、品質次元 (構文/意味的正確性、カバレッジ、一貫性、簡潔さ)、ルール/制約による品質確保、知識エンジニアに求められるスキル (技術 + コミュニケーション + 概念化能力 + 忍耐 + 組織スキル) を整理。LLM の登場でこれらが変わりつつある、という前振り。

### 2.2 Automation in KE (L186–274) — 推奨

KE 自動化の系譜:
- **知識抽出**: NER / entity linking / relation extraction
- **KG refinement**: 完全性 (link prediction, embedding) と修正 (logical inference, statistical relational learning)
- **KG enrichment**: オントロジー refinement / alignment
- **LLM 系**: pre-trained BERT/GPT、**fine-tuning と prompting** (zero-shot / few-shot)

「LLM が KE を支援する」と「KG が LLM を支援する」の 2 つの方向性のうち、**本研究は前者**。

### 2.3 Responsible AI (L276–410) — 推奨

AI 安全性の柱: **透明性 / プライバシー / 説明責任 / 公平性・バイアス**。LLM はブラックボックスで、訓練データの不透明性 → プライバシー・バイアスのリスクを増幅する。

**Data Cards / Model Cards 系のフレームワーク** が解決策の 1 つ (Mitchell et al., Pushkarna et al.) — メタデータ (詳細・意図された用途・性能・倫理的配慮等) を構造化記述する。後の **KG cards** 提案の伏線。

NER のバイアス、SRL の人口統計バイアス・常識バイアスなど、知識抽出パイプラインに混入するバイアスは多種多様であることも整理。

- 39 名 (15 機関、博士課程 31 名、ポスドク 2 名、講師 1 名、教授 2 名、業界 3 名)

## 4. Results (L562–995) — **必読**

### 4.1 LLM 利用時の主な課題 (L564–626)

3 つの課題が浮かび上がる:

**Dataset (データセット選定が最大の難所)**:
- 「最も困難なのはデータセットを見つけること」(参加者 8)
- ハッカソンの時間制約 + ドメイン知識のセットアップが圧迫
- 一方で「LLM がデータセット構築を助ける」期待もある (参加者 13)

**Prompting (プロンプティング)**:
- セマンティックウェブ系の人は NLP 経験が少なく、プロンプティングが難しい (参加者 1)
- プロンプトを反復テストする時間が大きい
- 「プロンプトテンプレート」のニーズ
- 一旦良いプロンプトに収束しても、**応答の一貫性**を保つのが厄介。スクリプト自動化時の構文エラー破綻も問題
- 「数十億パラメータのモデルは制御不能、数日後に再現できない」という根本的な懸念 (参加者 9)

**Evaluation (評価)**:
- KE タスク特有のベンチマーク不足 → 手動評価依存
- 「データセットは合成できても、評価は依然として手動が抜けない」(参加者 5)
- 制御されたデータセット (KG 構築) なら自動評価可能だが、競合質問・オントロジー設計はゴールドスタンダードがない
- HELM (Holistic Evaluation of Language Models) のような全体的ベンチマークの活用が提案

### 4.2 LLM 出力の評価方法 (L628–718)

**現行評価の限界**:
- F1/precision/recall + ゴールドスタンダード比較は単純タスクでは十分
- 階層関係などの構造ミスは「データの誤り以上に重大」(参加者 2)
- LLM が **追加で生成した知識** (例: 「Y は EU の一部」) と、**LLM が見落とした知識** の両方を評価する仕組みが必要
- LLM 出力は文字列でフォーマット不整合 (JSON 指定でも壊れる)、データ型混在 (日付・URL・画像)
- 現行の評価は「一致」前提で「類似性」を扱えない

**新しい評価手法の提案**:
- ゴールドスタンダードオントロジー集
- LLM オントロジーレビュー用ツールキット (OntoClean / OOPS の延長)
- 実シナリオで一貫性をテスト (クエリ性能、QPS、意味論一貫性)
- **ファクトチェック技術** (参加者 10)
- **敵対的アルゴリズム** (参加者 13) — BARD で機密データ流出を引き出した例
- **強化学習の self-play** (参加者 12) — 複数回答を比較してリファイン
- **人間の満足度** (参加者 3) — パフォーマンスではなくユーザー満足度を最適化

### 4.3 必要なスキル (L720–836)

**持っていて役立つスキル**:
- チーム内コミュニケーション (多分野グループでの効率)
- オントロジー構築スキル → プロンプトに文脈を追加できる
- **プロンプティングスキル** — 役割/ペルソナ → 短いタスク説明 → in-context learning の例 の構造を理解する
- コーディング / API / Hugging Face / PyTorch / Git
- 科学的フレームワーク (目的・仮説・評価戦略・実験設計)

**スキルギャップ**:
- オントロジー設計 (出力評価のため必須)
- 経験のないメンバーの一部は「プロンプティングはスキルではない、開発者なら誰でもできる」と評価 (参加者 7, 13, 11)
- 一方、「試行錯誤の先に行きたい」と希望するメンバーも多数 (参加者 4, 3, 1, 14)
- ハードウェア (GPU, クラスタ)、LLM ファインチューニング

### 4.4 バイアス意識 (L837–939)

**驚くほど低い意識レベル**:
- 「人間にもバイアスがあるからどうせバイアスがある」 = バイアスを考慮する必要がない、と暗黙に主張 (Observer 2)
- 「用語は知っているが、直接取り組んだことはない」(参加者 10)
- 「それは私の分野ではない」(参加者 4) — 安全性をエンジニアリングと別物として扱う認識
- 興味はあるが「取り組む機会がなかった」(参加者 5)

**部分的な認識**:
- 出所 / 選択性 / 性別バイアスの種類認識
- バイアス緩和は「商業製品では重要だが研究室では二の次」(参加者 12)
- LLM 内部のリソースが見えないため、プロンプト側で対処してもデバイアスの効果は不明
- バイアス除去とモデル性能のトレードオフ問題 (参加者 8)

**ユーザー共創型の緩和** (co-creating bias mitigation) のアイデアも提案。

### 4.5 LLM 採用への信頼 (L941–995)

**前向き派**:
- 時間短縮、専門家間コミュニケーション改善、経験不足を補える
- 「人間を置き換えはしないが、KE 活動を支援できる」
- ただし human-in-the-loop が必須

**懐疑派**:
- ハルシネーション、最新性の欠如、信頼度情報なしの予測出力
- 「テスト目的では使えるが、本番では使わない — どこが正しいかわからない」(参加者 6)
- 複雑なシナリオでは出力制御が困難

両立する見方として、**信頼できる KG が AI/LLM をサポートする** という逆方向の関係も指摘。

## 5. Discussion (L997–1195) — **必読**

### 5.1 LLM を KE に使うときの根本的な問題 (L1010–1062)

**生成 AI の出力の根本的な曖昧さ**:
- 結果の **再現不可能性** = 関係や類似性は「事実」ではなく「統計的確率」
- 科学的方法と相性が悪い

**有望な使い方**: 非構造化データから構造化データセットを「キュレーション」させる用途 → KE に親和的

**評価の悪化**:
- 既存の評価課題 (正確性 + 一貫性 + カバレッジ + 簡潔さ) が LLM 導入で複雑化
- Wikidata 同様、入力 (寄稿者背景) と出力の両方を評価する必要

**敵対的アルゴリズムの実例**: BARD に同じ単語を反復させてトレーニングデータをギガバイト単位で漏洩させた例 → 安全性テストとして機能する

**人間満足度の罠**: 「LLM の自然言語応答は感情があるように読まれ、結果が悪くてもプロセス成功と感じさせる」 → KE は感情より論理が必要なので問題

### 5.2 重要スキル: プロンプティング (L1063–1085)

- プロンプティングが重要かは意見が分かれるが、**結果として全 KE タスクで使われた**
- KE 分野に **新しいスキルセットの時代**
- KE タスク用の **プロンプトテンプレート** は実用価値あり
- **「Knowledge Prompting Engineer」** という新しい役割がスキルギャップを埋める可能性
- 10 年前のクラウドソーシング導入と類似の革新

### 5.3 重要スキル: バイアス (L1087–1112)

- バイアス緩和への関与は明らかに不足
- 「特定の専門家に任せる」モデルは限界、AI 構築者全員のトレーニングに統合すべき
- バイアスの源が「データ中心」と捉える参加者が多いが、**KG/オントロジー/LLM そのものにもバイアス**がある
- KE プロセスへの LLM 統合は新たな害 (例: BARD のプライバシー突破) を生む

### 5.4 KG Cards (L1113–1195) — **本論文のオリジナル提案**

危害レビューを実装する仕組みとして **KG cards** を提案。Data Cards (Pushkarna et al.) / Model Cards (Mitchell et al.) の延長。

**6 つのカード**:
| カード | 内容 |
|---|---|
| **Provenance** | HELM 風のテスト + Data Cards 風の説明 |
| **Construction** | グラフ構造の詳細 (データ処理の透明性) |
| **Knowledge** | ドメイン、対応言語、含まれる知識の範囲 (例: 2023 年までの英国首相) |
| **Completeness** | ノード数・関係数・平均リンク・クラス階層 (Heist et al. 風のネットワーク分析) |
| **Safety** | データ・適用・関係・公平性・説明可能性に関する安全チェック |
| **Licensing** | 構築データと結果グラフのライセンス |

Hugging Face の dataset cards / model cards との対応も整理 (Table 4-5)。KG embedding モデル (TransE 等) のドキュメンテーションへも拡張提案。

## 6. Conclusion (L1197–1259) — 推奨

LLM は KE に新時代をもたらす。現状は不安定さゆえに KG が信頼できる情報源として残るが、将来は不透明。

**スキル + ドキュメンテーション** の重要性:
- プロンプトテンプレート + KG cards
- 10 年前のクラウドソーシング導入と同様の革新になり得る

**Limitations**: 欧州研究所偏重、博士課程学生中心 (10年以上のシニアの意見が欠落)、3 日間で全 KE タスクは網羅できず

**Future work**: LLM が非専門家ユーザーの KG 開発を支援する方向、開発パイプライン全体でのバイアス特定・緩和の実用研究
